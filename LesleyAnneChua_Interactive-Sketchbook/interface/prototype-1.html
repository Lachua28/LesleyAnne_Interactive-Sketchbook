Structural Reflection

Structural decisions this week:

I decided to make a basic version of all my screens already. The biggest decision was to add a header bar at the top of both the mobile and desktop versions. For mobile, I’d make all the elements centred. For the desktop frames, I made the decision to use two columns for the landing page, but not the others. This is because I’d rather keep the white space on the right side of the page on text-heavy pages to account for window resizing. Alternatively, images could go in that space.

What changed:

My current site has some minimal CSS already, so this version has only some tweaks. The text size and weight hierarchy seems okay enough on my current site, so my prototype mimics the current CSS. The big changes are the header bar at the top and using two columns on the landing page.

What problem from your audit this addresses:

For all the pages, the main fix is making navigation accessible on every page via a (implied) menu. Because my micro-project is so self-contained, I feel that I cannot solve most of the heuristic problems in my audit without re-coding the P5. (Though, I may move the instructions outside of the P5 canvas so I can say the text that has to be within the canvas is modified.) I clarified hierarchy by pushing size and grouping a bit more. Also, I added some horizontal lines, which I think helps somewhat with hierarchy or content separation.

What still feels unresolved:

I’m unsure how to show “visible system status” because two of the prototypes are of static pages, and my micro-project is very self-contained. The system status within the micro-project feels relatively clear to me, but it’s not shown in the prototype for simplicity’s sake. There are currently no clear “consistent interaction cues,” but I plan on adding this by creating a Figma frame showing the side menus when they are open.
